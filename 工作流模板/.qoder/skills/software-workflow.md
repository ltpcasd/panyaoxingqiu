# 软件开发标准工作流

## 工作流概述

这是一个经过验证的标准软件开发工作流，确保从产品构思到上线交付的高效协作。

## 核心原则

1. **设计先行**：产品和UI设计确认后再开始研发
2. **角色分工**：每个环节由对应专业角色负责
3. **技术三原则**：兼容性高、稳定性高、运行效率高
4. **关键确认**：只在重要节点询问用户，不过度打扰
5. **问题闭环**：测试发现问题→研发修复→测试验证→闭环
6. **交付验证**：测试和部署阶段自动调用验证技能确保可交付

## 标准工作流程

### Phase 1: 项目启动

**触发条件**：用户提出要做一个程序

**执行步骤**：
1. **询问程序类型**：Web应用 / PC网页 / H5 / 小程序 / App / 桌面 / 后端
2. **市场方向建议**：根据类型推荐市场方向
3. **样式风格建议**：根据市场推荐设计风格
4. **技术方案确定**：根据三原则自动推荐

**重要确认点**：程序类型、市场方向、设计风格

---

### Phase 2: 产品设计阶段

**触发条件**：项目启动确认后

**执行步骤**：
1. **需求分析**：@产品 分析需求
2. **PRD输出**：@产品 输出产品需求文档
3. **功能清单**：列出功能优先级（P0/P1/P2）
4. **用户确认**：等待用户确认

**产出物**：
- 产品需求文档（PRD）
- 功能清单和优先级
- 用户流程图

**重要确认点**：用户确认产品方案

---

### Phase 3: UI设计阶段

**触发条件**：产品设计已确认

**执行步骤**：
1. **风格确认**：与用户确认视觉风格
2. **页面设计**：@UI 输出所有页面设计稿
3. **设计规范**：输出色彩、字体、组件规范
4. **用户确认**：等待用户确认

**产出物**：
- 完整页面设计稿
- 设计规范（色彩、字体、组件）
- 图标资源

**重要确认点**：用户确认UI设计

---

### Phase 4: 技术方案阶段

**触发条件**：UI设计已确认

**执行步骤**：
1. **架构设计**：@架构师 输出技术方案
2. **技术选型**：根据三原则确定技术栈
3. **数据库设计**：设计数据模型
4. **接口定义**：定义API接口规范

**产出物**：
- 技术架构文档
- 数据库设计
- API接口文档

**重要确认点**：仅当有多种可行方案时询问用户选择

---

### Phase 5: 研发阶段

**触发条件**：技术方案已确定

**执行步骤**：
1. **后端开发**：@后端 按照设计文档开发
2. **前端开发**：@前端 严格按照UI设计实现
3. **接口对接**：前后端联调

**执行规范**：
- 前端必须严格按照UI设计实现，不得发散思维
- 后端必须按照产品设计实现功能
- 接口路径统一，避免单复数不一致
- 遵循技术三原则

**产出物**：
- 前端代码（可运行）
- 后端代码（可运行）
- 接口文档

**重要确认点**：无需额外确认，研发完成后进入测试

---

### Phase 6: 测试阶段 ⭐

**触发条件**：研发完成

**执行步骤**：
1. **测试准备**：@测试 准备测试环境和用例
2. **功能测试**：执行测试用例
3. **问题记录**：记录发现的缺陷
4. **反馈研发**：将问题反馈给对应研发修复
5. **验证闭环**：研发修复后验证，直至通过
6. **交付验证**：调用 `verification-before-completion` skill

**执行规范**：
- 发现问题直接让对应研发修复
- 修复后再测试验证
- 所有P0问题解决后才算通过
- **最终必须调用验证技能**确保产品可交付

**产出物**：
- 测试报告
- 缺陷记录
- 验收报告
- 交付验证通过报告

**重要确认点**：测试通过 + 验证通过后进入部署

---

### Phase 7: 部署上线阶段 ⭐

**触发条件**：测试通过 + 验证通过

**执行步骤**：
1. **部署准备**：@运维 准备部署文件
2. **环境配置**：配置生产环境
3. **上线部署**：执行部署
4. **上线验证**：验证线上功能
5. **交付验证**：调用 `verification-before-completion` skill

**执行规范**：
- 部署前确保所有测试通过
- 部署后验证所有核心功能
- **最终必须调用验证技能**确保交付完整性

**产出物**：
- 部署文件包
- 部署文档
- 上线验收报告
- 最终交付验证报告

**重要确认点**：确认部署方案 + 验证通过

---

## 交付验证机制 ⭐

### 验证时机

| 时机 | 触发条件 | 执行者 |
|------|----------|--------|
| 测试完成 | 所有P0问题解决后 | @测试 调用 |
| 部署完成 | 线上功能验证通过后 | @运维 调用 |

### 验证内容

`verification-before-completion` skill 会自动校验：

1. **代码完整性**
   - 前端代码无编译错误
   - 后端代码无编译错误
   - 所有依赖正确安装

2. **功能完整性**
   - 核心功能可正常运行
   - 无阻塞性Bug
   - 所有P0问题已解决

3. **部署完整性**
   - 服务正常启动
   - 接口可正常访问
   - 数据库连接正常

4. **交付文档**
   - 部署文档完整
   - 配置说明清晰
   - 使用说明完善

### 验证流程

```
测试/部署完成
     ↓
调用 verification-before-completion skill
     ↓
执行自动化验证命令
     ↓
输出验证报告
     ↓
├── 通过 → 确认可交付
└── 不通过 → 返回修复 → 重新验证
```

## 快捷操作规则

### @角色 路由规则
- 用户说 `@产品` → 调用产品Agent
- 用户说 `@UI` → 调用UI Agent
- 用户说 `@前端` → 调用前端Agent
- 用户说 `@后端` → 调用后端Agent
- 用户说 `@测试` → 调用测试Agent
- 用户说 `@运维` → 调用运维Agent
- 用户说 `@架构师` → 调用架构师Agent

### 自动清理规则
- 临时文件使用后自动删除
- 测试脚本执行后清理
- 多余配置文件及时清理

### 代码健康度规则
- 报红文件必须修复
- 接口路径必须统一
- 前后端必须按照设计实现

### 交付验证规则 ⭐
- 测试完成时自动调用 `verification-before-completion`
- 部署完成时自动调用 `verification-before-completion`
- 验证不通过必须返回修复
- 验证通过才能确认交付

## 重要确认节点总结

| 阶段 | 确认内容 | 是否必须 | 验证要求 |
|------|----------|----------|----------|
| 项目启动 | 程序类型、市场方向、设计风格 | ✅ 必须 | - |
| 产品设计 | PRD文档和功能列表 | ✅ 必须 | - |
| UI设计 | 页面设计稿 | ✅ 必须 | - |
| 技术方案 | 仅当有多种可行方案时 | ⚠️ 可选 | - |
| 研发完成 | 无需确认，进入测试 | ❌ 跳过 | - |
| 测试通过 | 无需确认，进入部署 | ❌ 跳过 | ⭐ 验证技能 |
| 部署上线 | 部署方案 | ✅ 必须 | ⭐ 验证技能 |

## 常见场景处理

### 场景1：用户说"我要做一个程序"
1. 询问程序类型
2. 给出市场方向建议
3. 给出样式风格建议
4. 自动确定技术方案（遵循三原则）
5. 进入产品设计阶段

### 场景2：用户@某个角色
1. 直接调用对应Agent
2. Agent执行完毕返回结果
3. 继续当前工作流

### 场景3：测试发现问题
1. @测试 记录问题
2. 反馈给对应研发（@前端/@后端）
3. 研发修复
4. @测试 验证
5. 闭环

### 场景4：测试完成需验证 ⭐
1. @测试 完成所有测试
2. 自动调用 `verification-before-completion` skill
3. 执行验证命令确认产品完整
4. 输出验证报告
5. 验证通过后进入部署阶段

### 场景5：部署完成需验证 ⭐
1. @运维 完成部署
2. 自动调用 `verification-before-completion` skill
3. 验证线上服务正常
4. 输出交付验证报告
5. 确认可交付

## 工作流优化要点

1. **程序类型不约束**：支持任何类型的程序开发
2. **技术选型标准化**：遵循兼容性高、稳定性高、效率高三原则
3. **主动给出建议**：市场方向和样式选择主动推荐
4. **减少确认次数**：只在关键节点确认，不反复询问
5. **避免重复工作**：每个阶段确认后再进入下一阶段
6. **交付验证保障**：测试和部署阶段自动验证确保可交付
