## 一、Bug等级定义

### 1.1 严重等级说明

| 等级 | 名称 | 定义 | 示例 | 处理时效 |
|------|------|------|------|----------|
| P0 | 致命（Blocker） | 系统崩溃、数据丢失、核心功能不可用，阻塞测试进度 | 小程序启动崩溃、登录失败、数据丢失 | 立即修复 |
| P1 | 严重（Critical） | 主要功能异常，无临时解决方案，严重影响用户使用 | 配对失败、无法发送信件、图片上传失败 | 24小时内 |
| P2 | 一般（Major） | 功能缺陷有临时解决方案，或次要功能异常 | 某个筛选条件失效、部分文案错误 | 3天内 |
| P3 | 次要（Minor） | 对功能使用无影响的UI问题、文案优化 | 按钮对齐偏移、颜色差异、提示文案不清晰 | 下版本修复 |
| P4 | 建议（Suggestion） | 优化建议、用户体验改进 | 交互优化建议、动画效果优化 | 评估后处理 |

## 四、Bug生命周期

```
新建(New) → 确认(Confirmed) → 分配(Assigned) → 修复(Fixed) → 验证(Verified) → 关闭(Closed)
                ↓                    ↓
            拒绝(Rejected)       重新打开(Reopened)
```

### 状态说明

| 状态 | 说明 |
|------|------|
| 新建(New) | 测试人员新提交的Bug |
| 确认(Confirmed) | 开发确认是Bug |
| 拒绝(Rejected) | 开发确认不是Bug或无法复现 |
| 分配(Assigned) | Bug已分配给开发人员 |
| 修复(Fixed) | 开发已修复，待验证 |
| 验证(Verified) | 测试验证通过 |
| 重新打开(Reopened) | 验证失败，重新打开 |
| 关闭(Closed) | Bug流程结束 |

## 五、Bug处理规范

### 5.1 开发人员规范

1. **确认时效**
   - P0 Bug：立即确认
   - P1 Bug：4小时内确认
   - P2 Bug：1天内确认
   - P3/P4 Bug：2天内确认

2. **修复时效**
   - P0 Bug：立即修复
   - P1 Bug：24小时内修复
   - P2 Bug：3天内修复
   - P3 Bug：下版本修复
   - P4 Bug：评估后处理

3. **填写修复说明**
   - 描述修复方案
   - 关联代码提交记录
   - 说明影响范围

### 5.2 测试人员规范

1. **Bug提交要求**
   - 标题简洁准确
   - 复现步骤清晰完整
   - 必须附带截图或视频
   - 日志信息完整

2. **验证要求**
   - 验证原Bug已修复
   - 回归相关功能
   - 更新Bug状态

3. **关闭标准**
   - 验证通过
   - 或确认非Bug
   - 或确认延期处理
